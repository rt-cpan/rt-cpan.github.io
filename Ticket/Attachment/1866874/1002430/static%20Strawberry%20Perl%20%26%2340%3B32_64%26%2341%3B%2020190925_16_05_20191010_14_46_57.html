Advisories

M

SAST: AccessControl.Bypass  

Description

Attackers can issue a request directly to a page (URL) that they may not be authorized to access. If the access control policy is not consistently enforced on every page restricted to authorized users, then an attacker could gain access to and possibly corrupt these resources.

Make sure that the access control mechanism is enforced correctly at the server side on every page. Users should not be able to access any information that they are not authorized for by simply requesting direct access to that page.

Ensure that all pages containing sensitive information are not cached, and that all such pages restrict access to requests that are accompanied by an active and authenticated session token associated with a user who has the required permissions to access that page.

M

SAST: PrivilegeEscalation  

Description

A privilege escalation vulnerability is where an unprivileged user of a system gains privileged access to that system gaining sufficient rights to compromise, take control of, and ultimately destroy the entire system. The elevation of privilege threats include those situations in which the attacker has effectively penetrated all system defenses and becomes part of the trusted system itself.

Typically an attacker is able to elevate his or her privileges by taking advantage of (exploiting) a process that grants or uses higher privileges or system administrator privileges. A number of programs actually create processes as different users.

In the event of a program creating a process and running another program in that process with elevated privileges, the programmer must take care not to give rogue programs higher privilege. In the event of a programmer creating a process with lower privileges, the programmer needs to be certain to check the return codes of the process creation and privilege lowering operations for errors. If any of these operations fails, the program may be running at a higher privilege than anticipated and may perform certain unintentional operations as an elevated user.

Mitigation:

One way to prevent possible Privilege Escalation in applications is to design the application so privileged operations are separated. The notion of "Privilege Separation" is discussed in a number of white papers available on the Internet and elsewhere. Privilege separation is a generalized approach to the problem that lets a significant part of the application run without the need for elevated privileges. Programming errors that now occur in these unprivileged parts of the application can no longer be abused to gain unauthorized privileges. Privilege separation is orthogonal to "capability" or "role-based" security systems and may be used to enhance the security of such systems even further. Using such privilege separation techniques when designing and implementing applications will enhance the security of the application. An example of an application that was re-designed using privilege separation is the OpenSSHprogram. The separation of privileges in the OpenSSHapplication significantly reduced the amount of OpenSSH code that is executed with elevated privileges. It is known that the re-engineering work done for OpenSSH that implemented privilege separation would have prevented past privilege escalation vulnerabilities in the OpenSSH code base prior to the re-design.

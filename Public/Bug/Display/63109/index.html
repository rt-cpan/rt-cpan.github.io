Environment: Perl version: v5.12.0 Windows 32-bit DBD::ODBC version: v1.25 32-bit Environment: Virtual Windows Server 2008 64-bit Issue: DBD::ODBC seems to always rebind columns before each fetch - irrespective of whether a successful bind has already been made. This resulted in a memory leak whereby extracting a lot of rows from a simple table eventually exhausted all memory within the 2Gb 32-bit address space. On closer inspection of the DBD::ODBC code (dbdimp.c), controlling flag imp_sth->done_bind never seems to get set. Adding imp_sth->done_bind=1; at the bottom of bind_columns(), rebuilding and installing resolved the issue for me. Curiously, I haven't seen this behaviour on a Virtual Windows XP 32-bit environment suggesting that my hack above is not the whole story! Trace log snippet and calling code snippet below.

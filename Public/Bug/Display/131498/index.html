Subject: XML::LibXML::SAX stopped parsing &#38; in an attribute value

text/plain 2.1k
2.0202 stopped expanding external entities. But it also broke parsing internal &#38; entity that is located in an attribute value and when it is parsed with XML::LibXML::SAX. Reproducer: #!/usr/bin/perl use XML::LibXML::SAX; XML::LibXML::SAX->new(Handler => 'Handler')->parse_string(<<'END'); <?xml version="1.0"?> <root> <predefined param="&quot;">&quot;</predefined> <numeric param="&#65;">&#65;</numeric> <numericampersand param="&#38;">&#38;</numericampersand> </root> END package Handler; my $text = ''; sub start_element { my ($self, $node) = @_; print "STARTELEM name=$node->{LocalName}\n"; for my $key (sort keys %{$node->{Attributes}}) { printf "\tATTR name=%s value=%s\n", $node->{Attributes}->{$key}->{Name}, $node->{Attributes}->{$key}->{Value}; } $text = ''; } sub end_element { print "\tTEXT $text\n" if $text ne ''; $text = ''; } sub characters { my ($self, $node) = @_; $text .= $node->{Data}; } Output with old XML-LibXML: $ perl -I/home/test/fedora/perl-XML-LibXML/XML-LibXML-2.0201/blib/{lib,arch} ./saxent STARTELEM name=root STARTELEM name=predefined ATTR name=param value=" TEXT " STARTELEM name=numeric ATTR name=param value=A TEXT A STARTELEM name=numericampersand ATTR name=param value=& TEXT & TEXT Ouput with new XML-LibXML: $ perl -I/home/test/fedora/perl-XML-LibXML/XML-LibXML-2.0202/blib/{lib,arch} ./saxent STARTELEM name=root STARTELEM name=predefined ATTR name=param value=" TEXT " STARTELEM name=numeric ATTR name=param value=A TEXT A STARTELEM name=numericampersand ATTR name=param value=&#38; TEXT & TEXT See the difference in parsing <numericampersand param="&#38;">. Please note that XML::LibXML parser does not suffer from it. This bug causes RDF-Trine tests to fail <https://github.com/kasei/perlrdf/issues/167>. By the way I looked into XML::LibXML::SAX and XML::LibXML::SAX::Parser and I cannot see how a user can enable parsing external entities and DTDs when using XML::SAX. I think XML::LibXML::SAX could use a SAX feature for enabling it.

This was discovered by a RDF-Trine test suite failure <https://github.com/kasei/perlrdf/issues/167>. First, I suspected XML-LibXML <https://rt.cpan.org/Ticket/Display.html?id=131498>, but it turned out that the unexpanded entity comes from libxml2 C library itself. And that it's not just an ommitted expansion, but that it's actually an escape sequence for "&" characters. Other XML metacharacters (like "<") are not affeced. Also text nodes are also not affected. My finding was confirmed by an old libxml2 bug report <https://bugzilla.gnome.org/show_bug.cgi?id=316487>. This patch "fixes" this discepancy by replacing all "&#38;" subtstrings with a literal "&" in SAX interface of start_element() callbacks.

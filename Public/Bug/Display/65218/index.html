Perhaps this could just be clarified in the documentation, but my co-workers and I were wondering why this code, which expressly requests an 'A' record also includes a 'CNAME' record in its response. Shouldn't it just return an 'A' record? my $resolver = Net::DNS::Resolver->new(); my $packet = $resolver->search('dblack1.law.bepress.com', 'A'); foreach my $rr ($packet->answer){ print $rr->type,"\n"; } # prints # CNAME # A FWIW, "dig" has the same behavior, "dig dblack1.law.bepress.com A" returns two records, "dig dblack1.law.bepress.com CNAME" just returns the cname, same as Net::DNS. Is that behavior of search() a bug, or are we just misunderstanding something? If you could let us know, or point us to a better resource to ask, we'd much appreciate it. Thanks!

Dear Kevin, The zoneedit nameservers that serve bepress.com have a CNAME resource record for de name dblack1.law.bepress.com (pointing to dblack1.bepress.com.). No "A" resource record is associated with dblack1.law.bepress.com., but because you are doing the query via a recursive resolver (the service used for looking up the value), the algorithm described in section 4.3.2 of RFC1034 applies. I quote subitem 3.a: If the data at the node is a CNAME, and QTYPE doesn't match CNAME, copy the CNAME RR into the answer section of the response, change QNAME to the canonical name in the CNAME RR, and go back to step 1. You are observing normal recursive resolver behaviour. Best regards, Willem On Fri 28 Jan 2011 13:56:31, KGOESS wrote: Show quoted text

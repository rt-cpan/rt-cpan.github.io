Good news ! It's interesting to know that the API could run without the Perl >= 5.8.0 requirement. Unfortunately, a full success with the installation test suite is not sufficient to prove that every method in the package will be fully functional in every local environment. For practical reasons, the test suite operates with the default character set (i.e. iso-8859-1). For now, there is only one way to check the encoding functionality in a given local character set implies an interactive test procedure. The user must 1) explictly indicate the character set to be used (using the localEncoding function); 2) provide a character string including a typical set of local, non-ASCII characters, to be inserted in an OOo document; 3) inversely, provide an OOo document including typical characters to be extracted and displayed; 4) appreciate the results (i.e. say if the characters are correctly encoded in the two situations). Up to now, I've not tested Encode::compat. So, because of a lack of time and of users' reports from differents linguistic areas, I prefer to include a 5.8 requirement. But if you and others could send me positive reports about your 5.6.1 hack with a significant panel of encodings, and if Perl 5.6.1 remains a significant target, I could easily include the option as an optional patch in the next release. In any case, thank you for the information

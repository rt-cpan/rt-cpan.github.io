The "return if $@;" line should be more like if ($@) { (my $file = $package) =~ s/::/\//g; $file .= ".pm"; # ignore "Can't locate Foo.pm in @INC" # but warn for "Can't locate object module ..." warn "Warning: $@" if $@ !~ /^Can't locate \Q$file\E in/; return; } The docs should also mention that the functions are found by loading the module and what happens if the module can't be loaded. -- Tim Bunce
  Fri Sep 18 16:03:23 2009 FERRENCY [...] cpan.org - Correspondence added
I've just run into this same problem, but outside the context of XS. It seems sensible for Pod::Coverage to detect the difference between "can't compile the package" and "no public symbols defined." Pod::Coverage already reports "can't find pod" if the package can't be found at all, but all compilation errors are treated as "no public symbols defined. My recommendation is to have _get_syms return an arrayref on success, or undef on failure. Then coverage() can set why_unrated appropriately. Here are patches which implement this, as well as a simple test for it. Thanks!

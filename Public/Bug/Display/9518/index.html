text/plain 2.7k
'make test' was failing with a segmentation fault on packeb. Also, unpackeb('p5.2') was returning the wrong numbers. The problem turned out to be a conflict with a library function named pow10, which was causing pow10[ndec] to return garbage instead of the proper power of 10. This wouldn't have caused a segfault, except that CF_num2packed & CF_num2zoned don't check for numbers that are too big. If you do something like packeb('p16', 1.0e99); you get a segfault. The attached patch: 1) renames pow10 to pows_of_10 and makes it static. This fixes the actual problem I was having. 2) changes CF_num2packed & CF_num2zoned to check the size of the number they're about to pass to sprintf. If it's too big, they exit with croak instead of segfaulting. The patch also includes two new tests to make sure behavior 2 is working. Even with this problem, Convert::IBM390 has been really useful. Thanks.

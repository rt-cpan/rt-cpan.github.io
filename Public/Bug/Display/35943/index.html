Subject: ExtUtils::CBuilder fails to embed manifest when compiling Win32::IPC
When trying to compile Win32::IPC with Visual C++ 9.0 on ActivePerl I found that its tests were failing. Upon investigations I noticed that when embedding manifest with mt the manifest filename was incorrect. For example: mt -nologo -manifest "lib\Win32\IPC.dll.manifest" - outputresource:"blib\arch\auto\Win32\IPC\IPC.dll";2 Notice that the builddir for .dll and .manifest files differ. I don't know enough internals on CBuilder to determine what's the real issue here, but it appears that upon entering link in Windows.pm the lib_file and object files are in different directories, which causes such a strange situation. I currently have two solutions. The first one is a dirty hack to define $spec{manifest} as $spec{output}.'.manifest'. The second one is to actually tell link the proper manifest filename with -manifestfile switch. I'm attaching two respective patches.

I'd be inclined to go with the first solution--the 'dirty hack'. I don't think it really is all that 'dirty'. It's basically what ExtUtils-MakeMaker does. When you build an XS module with EU-MM the Makefile created contains an $(INST_DYNAMIC) target for which the recipe includes this line: mt -nologo -manifest $@.manifest -outputresource:$@;2 && del $@.manifest which simply defines the manifest file as output file with '.manifest' appended. (Actually, my first reaction was 'how can $spec{output} and $spec{manifest} have different paths anyway???' but of course it's the "||=" subtlety that I missed. $spec{manifest} gets set by a ||= line, but $spec{output} doesn't because it's already been set by the definition of $output and %spec a few lines earlier. Given that that *always* happens, I think it would be clearer to delete the '$spec{output} ||=' line as well.)

  Mon Oct 19 18:22:47 2009 JDB [...] cpan.org - Correspondence added
I notice that the code still uses the old check for explicit VC versions to decide whether to use the manifest file or not. For ExtUtils::MakeMaker we have already changed to this more robust versions: if exist $@.manifest mt -nologo -manifest $@.manifest -outputresource:$@;2 if exist $@.manifest del $@.manifest}); It will embed a manifest if it finds one. That way the code works correctly for the newer 64-bit VC versions that do require a binding manifest.

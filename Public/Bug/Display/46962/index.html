Subject: Moose coercion bug.
The change to the coercions mentioned here: 0.76 Mon, April 27, 2009 * Moose::Meta::TypeConstraint - Do not run coercions in coerce() if the value already passes the type constraint (hdp) Is lame. This breaks the previously **supported** behavior of: subtype 'OLStr' => as 'Str' ; coerce 'OLStr' => from 'Object' => via { $_=&overload_obj; $_=&trim_whitespace($_) } => from 'Str' => via { $_=&trim_whitespace($_) } ; I'm not arguing that changing the behavior to the non-backwards-compat behavoir isn't a good thing or wasn't called for. Just that the hacks that people have abused the Type system for, were the supported method of doing this, (like trimming whitespace) and that they are no longer functioning in ways that might create subtle bugs to the user. Here is my suggestion, if there isn't a ``where {}'' clause on the subtype, run the coercion anyway.

This is stupid, if you do this then at the least throw a compile time error anytime you coerce from a basetype to the same basetype or a parent of it. subtype 'OLStr' => as 'Str' ; coerce 'OLStr' => from 'Str' => via { $_=&trim_whitespace($_) } See the Str in `as`, and `from`. That is obviously an error (as he coercion won't ever trigger) that should be detected if you maintain the current retarded course. My bet is, if you detect that error you'll unleash the fury of hell that is currently cloaked in silent failures. The current retarded course doesn't stop there, look at this code class_type 'MooseClass'; subtype 'WTFObj' => as MooseClass ; coerce 'WTFObj' => from 'Any' => via { die 'HALLO NEWB' }; Guess what... Not going to run, or any other coercion from a parent type for that matter.. Let's see how many users get this one right. Send in an object, and it works, send in a string, and the coercion is never triggered..

I've also encountered this problem, and would like to add some additional information. I've successfully compiled both 32 and 64 bit versions of mysql-5.1.59, DBI-1.616, and DBD- mysql-4.020 on AIX 6, Solaris 10, and RHEL5, for all of perl5.10.1, 5.12.4, and 5.14.2. It passes the test suite successfully for all of those combinations except for the 64bit builds on both AIX and Solaris. The test suite failures are identical for all 6 cases (64bit AIX and Solaris, for all 3 major perl releases). In addition to the test failure reported here, I also get one additional failure: [2011-11-15 18:32:45] # Failed test at t/40server_prepare.t line 67. [2011-11-15 18:32:45] # Structures begin differing at: [2011-11-15 18:32:45] # $got->[0][0] = '0' [2011-11-15 18:32:45] # $expected->[0][0] = '101' [2011-11-15 18:32:45] [2011-11-15 18:32:45] # Failed test at t/40server_prepare.t line 80. [2011-11-15 18:32:45] # Structures begin differing at: [2011-11-15 18:32:45] # $got->[0][0] = '4294967296' [2011-11-15 18:32:45] # $expected->[0][0] = '1' [2011-11-15 18:32:45] # Looks like you failed 2 tests of 27. [2011-11-15 18:32:45] t/40server_prepare.t ........ [2011-11-15 18:32:45] Dubious, test returned 2 (wstat 512, 0x200) [2011-11-15 18:32:45] Failed 2/27 subtests Both of these tests are clearly related to how integers are handled in 64bit perl. The first failure is here: #bind test values ok($sth2->bind_param(1, 101, DBI::SQL_INTEGER), "binding int"); ... ... is_deeply($dbh->selectall_arrayref('SELECT * FROM t2'), [[101, 102, 103, 104]]); The code binds "101" to that parameter as a DBI::SQL_INTEGER, and something goes wrong. The second failure is similar, and occurs here: ok($dbh->do(q{CREATE TABLE t3 (id INT, mydata LONGBLOB)}), "creating test table"); my $sth3; ok($sth3 = $dbh->prepare(q{INSERT INTO t3 VALUES (?,?)})); ok($sth3->execute(1, 2), "insert t3"); is_deeply($dbh->selectall_arrayref('SELECT id, mydata FROM t3'), [[1, 2]]); Again, the code's binding a "1" to an INT, and something goes wrong. FWIW, DBI-1.616 passes it's entire test suite on all of the platforms described above. Observations: (1) This is independent of the perl5 core release, with identical failures for 5.10.1, 5.12.4, and 5.14.2. (2) This is almost platform independent (no problems on 64bit RHEL 5), with identical errors on AIX 6.1 and Solaris 10. This bug is clearly related to how 64bit integers are handled, but I have no idea where to look next. I hope this helps.

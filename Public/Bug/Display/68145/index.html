Hi, My response has not been so quick this time. I'm still haveing some difficulty trying to understand what you are trying to accomplish. I have not tried to combine things in this way before. "Because all the tied methods block, they should ALWAYS be used with timeout settings and are not suitable for background operations and polled loops. If a B<sysread> times out after receiving some characters, the actual elapsed time may be as much as twice the programmed limit. If no bytes are received, the normal timing applies." The examples did not have timeouts set. And used polled loops. Hi, There is no real problem with missing characters. The problem is in what you expect a tied filehandle to do. I've attached a subset (pc2a.pl) of your example serial_bugtest_pc2.pl that reads from a real filehandle (STDIN) rather than a serial port. That eliminates the module and the need for two independent ports to test. Start it, and then enter text...you'll need an abort (i.e. Control-C) to exit the loop. When I entered: "12\n34\n56\n78"...and eventually ^C since there is no timeout I saw the same behavior you reported initially. If you did have a timeout set, you could have retrieved the "78" using the lastlook method. But that doesn't really work well on a blocking call like you have in the tied filehandle case. What you wanted was the behavior you would get from a perl eof() function - but that doesn't work on terminals (or serial ports). My intent was to mimic the perl readline() function which is how regular filehandles are implemented. -bill BTW: neither $limit nor $count do anything useful in your example. The <FH> won't return until it finds a line terminator. And a comparison between a string and /(.*)/ will always match. But you don't need $1 since in that context is is always equal to $line.

Subject: WebService::Solr::Query::stringify removes special operands from arrayref
I believe I've found a bug where any call to stringify (really its _dispatch_value doing it) that involve parameters including a special operand will remove said operand from the arrayref on the first usage, rendering any further usage of the object to give an error of "Not a SCALAR reference at at Query.pm line 152". The issue occurs at line 106 with "shift @$v;" where _dispatch_value handles the special operands. It is being done so the operand isn't in the array when it is later looped upon for the rest of the parameters. An example would be debugging with $query->stringify to see the output of the query prior to passing it into $solr->search($query); or $solr- Show quoted text
>search($query->stringify). The first stringify will return the correct
query string, but the latter calls will fail due to no longer having '- and' in the arrayref. While there is obviously the work around of just storing the first call of stringify into a separate variable and using that moving forward, I think this should be fixed since the expectation of the stringify method is to just return a string, not actually modify the contents of the object's parameters. I have attached a patched Query.pm, though feel free to adjust it to your liking.

Syntax wise, it attempts to be as close to L<SQL::Abstract> WHERE clauses as possible, with obvious exceptions for idioms that do not exist in SQL. Just as values in SQL::Abstract are SQL-escaped, this module does the appropriate Solr-escaping on all values passed to the object (see C<escape()>). =head1 QUERY SYNTAX =head2 Key-Value Pairs The simplest way to search is with key value pairs. my $q = WebService::Solr::Query->new( { foo => 'bar' } ); # RESULT: (foo:"bar") =head2 Implicit AND and OR By default, data received as a HASHREF is AND'ed together. my $q = WebService::Solr::Query->new( { foo => 'bar', baz => 'quux' } ); # RESULT: (foo:"bar" AND baz:"quux") Furthermore, data received as an ARRAYREF is OR'ed together. my $q = WebService::Solr::Query->new( { foo => [ 'bar', 'baz' ] } ); # RESULT: (foo:"bar" OR foo:"baz") =head2 Nested AND and OR The ability to nest AND and OR boolean operators is essential to express complex queries. The C<-and> and C<-or> prefixes have been provided for this need. my $q = WebService::Solr::Query->new( { foo => [ -and => { -prohibit => 'bar' }, { -require => 'baz' } ] } ); # RESULT: (((-foo:"bar") AND (+foo:"baz"))) my $q = WebService::Solr::Query->new( { foo => [ -or => { -require => 'bar' }, { -prohibit => 'baz' } ] } ); # RESULT: (((+foo:"bar") OR (-foo:"baz"))) =head2 Default Field To search the default field, use the C<-default> prefix. my $q = WebService::Solr::Query->new( { -default => 'bar' } ); # RESULT: ("bar") =head2 Require/Prohibit my $q = WebService::Solr::Query->new( { foo => { -require => 'bar' } } ); # RESULT: (+foo:"bar") my $q = WebService::Solr::Query->new( { foo => { -prohibit => 'bar' } } ); # RESULT: (-foo:"bar") =head2 Range There are two types of range queries, inclusive (C<-range_inc>) and exclusive (C<-range_exc>). The C<-range> prefix can be used in place of C<-range_inc>. my $q = WebService::Solr::Query->new( { foo => { -range => ['a', 'z'] } } ); # RESULT: (+foo:[a TO z]) my $q = WebService::Solr::Query->new( { foo => { -range_exc => ['a', 'z'] } } ); # RESULT: (+foo:{a TO z}) =head2 Boost my $q = WebService::Solr::Query->new( { foo => { -boost => [ 'bar', '2.0' ] } } ); # RESULT: (foo:"bar"^2.0) =head2 Proximity my $q = WebService::Solr::Query->new( { foo => { -proximity => [ 'bar baz', 10 ] } } ); # RESULT: (foo:"bar baz"~10) =head2 Fuzzy my $q = WebService::Solr::Query->new( { foo => { -fuzzy => [ 'bar', '0.8' ] } } ); # RESULT: (foo:bar~0.8) =head2 Boost my $q = WebService::Solr::Query->new( { foo => { -boost => [ 'bar', '2.0' ] } } ); # RESULT: (foo:"bar"^2.0) =head2 Literal Queries Specifying a scalar ref as a value in a key-value pair will allow arbitrary queries to be sent across the line. B<NB:> This will bypass any data massaging done on regular strings, thus the onus of properly escaping the data is left to the user. my $q = WebService::Solr::Query->new( { '*' => \'*' } ) # RESULT (*:*) =head1 ACCESSORS =over 4 =item * query - stores the original query structure =back =head1 METHODS =head2 new( \%query ) Creates a new query object with the given hashref. =head2 stringify( ) Converts the supplied structure into a Solr/Lucene query. =head2 escape( $value ) The following values must be escaped in a search value: + - & | !

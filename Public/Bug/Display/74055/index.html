Report: in the current version (0.185) of Email::Valid, it uses an eval block to call the isa() method on a scalar that may not be an object. Under most circumstances, this will cause no visible errors, but when SIG{"__DIE__"} is assigned to a custom subroutine, the error in the eval block will still be caught, when no error should be generated/reported. Such invalid call to isa() seems to occur only when the username or the domain part starts with digits (such as 123@abc.com or abc@123.com) and the mxcheck option is enabled. I have fixed this by replacing the eval blocks with a preliminary verification call to Scalar::Util's blessed().

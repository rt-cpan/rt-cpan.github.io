> > On Oct 24, 2013, at 4:04 AM, Kurt Blackwell <kurt@blackwell.id.au > <mailto:kurt@blackwell.id.au>> wrote: >
>> However ProhibitExplicitReturnUndef encourages you to return empty >> lists from scalar functions which often causes me dramas.
> > I think this is one of those policies that has too many legit anti-cases > to make it useful. Lately I've been bitten by this: > > my $args = { > foo => 17, > user => $foo->user, > bar => 28, > }; > > and because $foo->user() returns empty where we have no user set yet, > I'm assigning a 5-element list to a hash, and the only way around it is > to call scalar $foo->user, or I can drop the policy from my perlcriticrc > and let user() return undef explicitly, and I think that's what I'm > going to be doing.
Or you can disable it locally for user() and document why, so you remember a year or so from now when debugging the code :-). I've been promoting Perl::Critic heavily at my company, and I explain to people that yes, you *can* turn off a particular policy if you have a need to word around it. I do that fairly often. But I also document *why* I'm doing it. We have a rigorous code-review process, and if I see a "## no critic(...)" without a (good) reason why, I'll bounce it back to the programmer.

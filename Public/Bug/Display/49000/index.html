Subject: [PATCH] ExtUtils::CBuilder::Platform::Windows::GCC - write_linker_script()
Hi, I have experienced a "linking problem" on Win32/strawberry perl (it uses mingw gcc compiler/linker). The problem appears when a module wants to specify some additional 'extra_linker_flags' and wants to use a linker_script. Example: *) we set 'extra_linker_flags' to "-Lc:/path/libs -lmylib1 -lmylib2" *) the generated linker command looks like this (simplified): g++.exe -o "blib\arch\auto\MyMod\MyMod.dll" ... -mdll -s -L"c:\\strawberry\perl\lib\CORE" -Lc:/path/libs -lmylib1 -lmylib2 "MyMod.lds" "MyMod.exp" *) linking fails with many errors about unknown references from MyMod.o to functions from libs specified in 'extra_linker_flags' (of course those function exists) *) it turned out that the position of the linker script "MyMod.lds" in the parameter order is crucial as this scripts contains also name of linked object (MyMod.o). What causes the problems is IMHO the fact that in linker parameters we specify: FIRST - objects (inside *.lds); NEXT - libraries set by 'extra_linker_flags' - so linker params goes in fact in this order: ... -Lc:/path/libs -lmylib1 -lmylib2 MyMod.o Unfortunately at least with mingw linker on Win32 it is quite significant what linker params order you choose: ... MyMod.o -lmylib1 -lmylib2 (1.objects 2.libraries - WORKS NICE) ... -lmylib1 -lmylib2 MyMod.o (1.libraries 2.objects - MAKES TROUBLES) After I have changed the linker parameters order to this: g++.exe -o "blib\arch\auto\MyMod\MyMod.dll" ... -mdll -s -L"c:\\strawberry\perl\lib\CORE" "MyMod.lds" -Lc:/path/libs -lmylib1 -lmylib2 "MyMod.exp" ... linking was finished without errors (I put linker script "MyMod.lds" with object definitions before libs specified by 'extra_linker_flags') To achieve this I have patched ExtUtils::CBuilder::Platform::Windows::GCC in the following way: ************* --- ExtUtil\CBuilder\Platform\Windows.pm +++ ExtUtil\CBuilder\Platform\Windows.pm.new @@ -647,7 +647,8 @@ @{delete $spec{perllibs} || []}, ) . ")\n"; - push @{$spec{other_ldflags}}, '"' . $script . '"'; + #it is important to keep the order 1.linker_script - 2.other_ldflags + @{$spec{other_ldflags}} = ( '"' . $script . '"', @{$spec{other_ldflags}} ); return %spec; } ************* I admit that different behaviour of mingw linker with different parameters order is still for me a "dark magic" but the basic idea of my patch - "put objects before libs" - is quite clear. I would appreciate if you could consider applying the above mentioned patch. Of course I may be wrong and the described issue can be solved in different way. -- kmx

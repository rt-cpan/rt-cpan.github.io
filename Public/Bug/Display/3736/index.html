Subject: lack of PL_FILTER to compliment PM_FILTER
While there is the PM_FILTER facility, there is nothing for .pl files along these same lines! I needed to make a package that customizes the pm _and_ pl files during the pm_to_blib() phase and found a dead end, so I duplicated the pm_filter setup for pl_filter appropriately. The patch attached implements the PL_FILTER feature and documents it exactly as PM_FILTER has been documented (in fact I literally duped and then s/PM/PL/gi as nessary). I would really like to see this feature implemented upstream as I'd hate to have to get everyone who wants to use my package patch their EU::MM just to build it properly.

> This wishlist item is for the compliment to PM_FILTER... PL_FILTER. This > option would allow Perl scripts to be filtered in the same ways the Perl > modules can be filtered (with a stream-type shell command). While .PL > files are extremely usefull in most cases. However, there are some cases > where only a few things need to be changed in the Perl scripts and > making each .pl into a .PL would introduce more code that just get's > eliminated in the end.
Because supplying raw snippets of shell programs is extraordinarily unportable, use of PM_FILTER is unofficially discouraged. I'd love to officially discourage it but I have no replacement. As PL_FILTER parrots the flawed PM_FILTER interface I unfortunately must reject it. I'd encourage you to work on filtering system for Module::Build instead. If you really want a PL/PM filtering mechanism for MakeMaker, I would accept a new filtering mechanism that took Perl code rather than shell snippets. Something to be fed to a perl -i -p maybe: FILTER_PM => 's{$$YOUR_CC}{$(CC)}' or perhaps something using STDIN and STDOUT: FILTER_PM => 'while(<>) { s{$$YOUR_CC}{$(CC)}; print }' Internally, one liners can be generated from this code and made portable using oneliner().

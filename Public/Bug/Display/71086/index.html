On Fri Oct 21 03:48:39 2011, ABRAXXA wrote: Show quoted text
> No, I mean indirect object creation syntax: > > my $ip = new NetAddr::IP::Lite '127.0.0.1';
I know what you mean, however this is purely an issue of style. The syntax works in either of the cases and is and will always be supported in perl. Whether I like it or you like it does not change its "correctness". Who are we to tell others how to write their code. I have a colleague that supervises other engineers that does not like "tabs" in code. His opinion on the matter is pretty strong. I happen to disagree and use tabs freely in my code and both of us get along just fine and the compiled objects work just fine. I think the same can safely be said for your preference for one syntax structure over the other. Perl does not care and code written one way or the other will run exactly the same on all platforms. I respect your opinion and I have been polite and included your preferred syntax in the NetAddr::IP documentation. That does not make the other syntax "wrong". It is merely your preference. Personally I prefer the "wrong" syntax because it is not as many keystrokes and no upper case an 4th row characters when writing code i.e. ->() so the typing is a bit faster. As I noted in my previous message, the syntax is used freely in other subroutine calls (i.e. import) in current perl documentation. Whether or not an "object" is being created (as in the case of "new" or not is not really the issue. A subroutine call is a subroutine call. Perhaps (humor intended) we should eliminate the "new" method and just call it "generateIP" and provide the instructions that say: ... this is similar the Exporter's import subroutine call. For example: $success = import packagename LIST $ip = generateIP NetAddr::IP LIST This particular syntax is EVERYWHERE in the perl documentation. The point I am making about "object" creation is that it is a fetish of "OOP" programers. There are actually people in the world that don't care one way or the other about "objects" since they are entirely imaginary anyway. The oop is a tool to help people conceptualize a simple idea that they otherwise would have difficulty understanding. It is similar to the concept of describing multidimensional space. Some people have difficulty with anything beyond 3 dimensions. What is remarkable about that is how simple it is to describe. The description of N space can be written in a small space on a single sheet of paper as an N x N matrix and problem in that context can be solve with simple linear algebra. Something a high school kid can accomplish if it is just explained to them in that fashion. Perhaps we can agree that is not necessary to force people to use OOP syntax when it is entirely unnecessary to do so with respect to the outcome of their efforts, usability, run-ability, and portability of their code. We can agree that it might look ugly, but so do a lot of "very compact" statements that can be written using perl syntax, particularly with methods that return blessed objects. Best regards, Michael

Subject: POE::Component::IRC::Plugin::NickReclaim seems to be broken
my perl version is: "This is perl, v5.8.8 built for i486-linux-gnu-thread-multi" Using POE::Component::IRC version 5.40 with PoCo::Plugin::NickReclaim version 1.0 installed with ``cpan'' I am not sure how this bug got into the module, or whether it's an actual bug or just something on my part. Basically, the it doesn't reclaim the nick at all. Looking at the code and dumping the values with Data::Dumper I see that: return PCI_EAT_NONE if $irc->nick_name() eq $irc->{nick}; in S_433 returns prematurely because ->nick_name() has the same value as $irc->{nick} when starting up. Also, if telling the bot (while already connected and on the channel) to change the nick to an already taken one the plugin doesn't "reclaim" it either, because $irc->{nick} is again the same as $irc->nick_name(); Anyway, I patched up the module (see attached file) works fine for start up and "asking to change the nick during running" scenarios. I am not sure if it is how you intended the module to work. But my patched version only tries to reclaim the "real" nicks, i.e. the ones with no underscore appended, i.e. if foo, foo_, and foo__ are taken it will change to foo___ and will poll only to see if foo is available. I think that cuts down on traffic a bit.

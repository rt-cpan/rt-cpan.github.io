Well, the real problem here is that D::P doesn't respect ANSI_COLORS_DISABLED. Furthermore, it even doesn't preserve it's value inside a call to _data_printer. Even for it's own test. Like the following: 1) A_C_D gets set/unset at the beginning of the test 2) then it get overwritten in 1st call to _data_printer to whatever it thinks is suitable 3) ..and is overwritten inside each call to _data_printer I've tried some straightforward patches (for A_C_D checks only), but they tend to break things on *nix. So real solutions seams to be a wrapper to colored(), that checks internal state in $p (precomputed) and never touches A_C_D. And run tests that are written using A_C_D=undef using {colored=>0} instead.

Yes. I felt a bit on the fence about how to behave regarding colors, and this particular behavior has changed a lot over the first few versions, but your statement pretty much covers it for me: there *is* a variable to activate/deactivate colors and Data::Printer *should* respect it. The patch to 0.20 should cover it, and it also fixes the Windows issue. It's somewhat based in your own colorization patch, but keeps using (and respecting) the environment variable unless you force colors with colored => 1. The default is still colored => 'auto' which from now on should do the right thing :) Thanks to everyone involved. I was able to successfully install it on a 64-bit Windows 7 machine.

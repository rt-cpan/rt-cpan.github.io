I've attached a simple example fix for the fact that the os/*.c code in Proc::ProcessTable does not use warn(). It turns out that (in an unrelated problem) Proc::ProcessTable is hit by the same problem that procps was on Linux systems (get more info on this at http://www.ussg.iu.edu/hypermail/linux/kernel/0202.2/0446.html) This leads to spurious warning on some of our systems which cannot be shut off using $SIG{__WARN__}. Fixing the larger problem of /proc/stat interaction would be good (though it's not tuely Proc::ProcessTable's problem), but making it not issue warnings to stderr without calling warn() is useful regardless since there are many instances (e.g. CGI, monitoring systems, etc) where calling warn() is useful for reporting to a human being, but simple stderr is likely to be thrown away.
